# Sean's RAW Editor

A client-side RAW/DNG photo editor for Panasonic RW2 and DNG files (works with any LibRaw-decodable format). Browse a folder of RAW files, pick one to adjust, tag it with keywords, see where it was shot, and export it. One image is open in the editor at a time. No backend; everything runs in the browser, and the RAW files themselves are never modified.

See [CHANGELOG.md](CHANGELOG.md) for the full history of what's been built and fixed.

## Stack

- Vite + React + TypeScript + Tailwind
- `libraw-wasm` (LibRaw compiled to WebAssembly) running in a Web Worker for RAW decoding
- WebGL2 fragment shader for the real-time, non-destructive edit pipeline
- `zustand` for state (edit params with gesture-grouped undo, crop tool state)
- Deployed to GitHub Pages via GitHub Actions on push to `main`. Live at sb1947jp.github.io/seans-raw-editor

## Architecture

- `src/lib/rawDecoder.ts` — wraps libraw-wasm. `decodePreview()` decodes at half resolution for fast interactive editing; `decodeFull()` decodes full-res, used only on export. `probeFile()` is the cheap third path used by the file browser: metadata plus the camera's *embedded* JPEG preview, no demosaic at all. GPS comes from LibRaw's `gps_data` and is converted to signed decimal degrees, deliberately strictly — `gpsparsed` distinguishes "no GPS" from a real 0°,0°, and a `'V'` (no-fix) status, a missing hemisphere ref or an out-of-range value is rejected rather than guessed at. Faithful decode settings: camera white balance, camera color matrix, no auto-brightness. Highlight reconstruction: if the frame contains clipped pixels, a second LibRaw blend-mode decode runs and its recovered data is spliced into *only* the clipped pixels (re-anchored via an empirically measured scale factor, C1 shoulder into the top ~20 levels); unclipped pixels stay bit-exact and unclipped frames pay nothing. Plain `highlight: 2` can't be the primary decode — it darkens the whole image by a per-image WB-dependent factor.
- `src/gl/renderer.ts` + `src/gl/shaders/adjust.frag.glsl` — the edit pipeline. One shader does exposure, highlight rolloff, white balance, brightness, contrast, highlights/shadows/whites/blacks, saturation/vibrance, sharpen, in that order. `EditCanvas` (interactive) always renders the full frame (`applyCrop={false}`); crop is only baked in at export time.
- `src/lib/whiteBalance.ts` — computes a 3×3 chromatic-adaptation matrix (CAT16/von Kries in LMS cone space) from Temperature/Tint, uploaded to the shader. Not naive per-channel gains — see CHANGELOG for why.
- `src/lib/filmStocks.ts` — "Film emulation" presets (Color section) bundling temperature/tint/saturation/vibrance/contrast to emulate specific film stocks.
- `src/lib/autoCrop.ts` — largest-inscribed-rectangle math for auto-cropping rotated images to avoid edge-clamp smudging; `fitAspectInRect` keeps a locked crop ratio while rotating.
- `src/lib/autoLevels.ts` — Auto Levels solver. Ports the shader's actual tone math into JS and numerically solves for exposure/blacks. **Must be kept in sync with the shader** — this drifted out of sync once already and produced bad results (see CHANGELOG).
- `src/lib/palette.ts` — `UI_COLORS` is the interface's whole colour vocabulary, expressed by role: `accent` (active/selected/live), `danger` (destructive, plus the map pin where red is cartographic convention), `heading` (neutral section titles), `muted`. The panel previously carried eight competing accents and the chrome started fighting the photograph. **Use `UI_COLORS`, not `JAPANESE_PALETTE` directly** — the full palette is kept only as a reference, and reaching into it is how a ninth colour sneaks back in. The histogram's R/G/B channel colours are exempt: that's data, not chrome.
- `src/state/editParams.ts` — single source of truth for all slider values, with `beginChange()`/`set()` gesture grouping so a whole drag or a whole preset-apply is one undo step.
- `src/state/cropTool.ts` — crop ratio/orientation UI state, separate from the crop rect itself (which lives in editParams).
- `src/components/FileBrowser/` — the **Files** tab of the editing panel: folder/file picker plus drag-drop, thumbnail list, keyword filtering, and the map. It renders content only, with no panel chrome of its own — `Sidebar` owns the frame, the scrolling and the Edit/Files tab bar (`sidebarTab` in `uiMode`). It was briefly a second, separate left-hand panel; folding it into a tab keeps the window to one column of chrome beside the photo. Thumbnail probes run strictly one file at a time — each one spins up a LibRaw wasm worker, so fanning out across a big folder would try to hold that many wasm heaps at once.
- `src/components/FileBrowser/MapView.tsx` — a hand-written slippy map over OpenStreetMap raster tiles (Web Mercator tile math, drag-to-pan, zoom buttons). Deliberately not Leaflet: the job is a grid of `<img>` tiles, and a map library would add a dependency, a stylesheet and marker assets that all need base-path wrangling for the Pages subdirectory deploy. OSM's tile policy requires the visible attribution, so don't remove it.
- `src/state/library.ts` — the browsed file list (session-only; `File` objects can't survive a reload), selection, and keyword filtering. Items are keyed by `name:size:lastModified` so re-picking the same folder dedupes.
- `src/lib/keywordStore.ts` — keyword tags in IndexedDB, keyed by **file name** so they survive reloads even though the `File` objects don't. The trade-off is that identically-named files in different folders share tags.
- `src/lib/idb.ts` — one shared IndexedDB handle for the session slot and the keywords. A single database name can only be open at one version at a time, so two modules each calling `indexedDB.open()` with their own version would deadlock on upgrade. The DB name is still `lumix-raw-editor` on purpose: renaming it would orphan every existing user's stored session.

## Working conventions (established over this project)

- **Never commit or push without being explicitly asked.** Make changes, verify them, report what changed, and wait. When asked to push, `git add` the specific files touched (not `-A`), write a real commit message explaining *why*, push, then confirm the GitHub Actions deploy succeeded (`gh run watch ... --exit-status`) and that the live bundle hash matches.
- **Verify color/tone changes with pixel readback, not just visual inspection.** Hook `gl.drawArrays` on the live WebGL context to capture `gl.readPixels()` after each render, then assert on saturation ratios, channel-clip counts, luma — this caught real bugs (hue-shifting clamps, oversaturated tint, contrast solving to 95) that a screenshot alone would have missed.
- **Test with real RAW files** — `/tmp/rawtest/` was the original fixture dir but no longer exists; there are real RW2 files under `~/Pictures` (e.g. the Australia/Koalas folder, which is geotagged and so is the fixture to use for GPS work). Copy into `public/`, reload, drag-drop via a synthetic `DragEvent`, wait for `canvas` to appear, then interact. Clean up with `rm -f public/<the test files>` — **not** `rm -rf public`, which would delete the tracked `public/favicon.svg`.
- **Synthetic `File` objects get a fresh `lastModified` every time they're constructed**, so re-dropping "the same" file in a test creates a duplicate library row. That's a test artifact, not a dedupe bug — real picker/disk files carry a stable mtime. Pass an explicit `lastModified` when testing dedupe.
- **Always run `npx tsc -b` and `npm run build` before considering a change done**, then remove build artifacts (`dist`, `vite.config.d.ts`, `vite.config.js`, `*.tsbuildinfo`) since they're gitignored but clutter the working tree if left from a manual build.
- **When the shader's tone pipeline changes, check `autoLevels.ts`** for whether its ported math needs updating too.
- **There's a strict Content-Security-Policy in `index.html`.** Adding any external resource (a font, an API, an analytics script, another tile provider) means widening it, and the app will silently half-work if you don't — a blocked stylesheet renders as unstyled Times, a blocked script leaves `#root` empty. `connect-src 'self'` is the important one: it means photo data has nowhere to be exfiltrated to, so think hard before loosening it. Verify CSP changes against a **production** build, not dev — dev serves everything from one origin and hides the problem.
- **`npm run preview` is the only local way to test the real deployed paths.** It passes `--base=/seans-raw-editor/` because `vite.config.ts` only applies that base when `command === 'build'`, and `vite preview` runs as `serve` — without the flag it serves at `/` while the built HTML points at `/seans-raw-editor/`, so every asset 404s into the SPA fallback and you get a blank page that looks like a code bug.
